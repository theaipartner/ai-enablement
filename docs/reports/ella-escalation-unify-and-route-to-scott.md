## What I did, in plain English

Built one shared escalation-fan-out helper and pulled both Ella escalation paths through it. The passive path previously DMed only the channel's primary CSM and never wrote an `escalations` row; now it writes the row via the existing `agents.ella.escalation.escalate()` wrapper AND fans DMs out to Scott (`ESCALATION_RECIPIENT_SLACK_USER_ID`) plus the primary CSM with dedup when those are the same person. The reactive path previously posted an in-channel ack with an `<@advisor>` Slack mention and never fired a backend DM; now the mention is gone (Sonnet's prompt edited to drop it) and a backend DM fans out to Scott + primary CSM after the escalations row is written, matching the passive shape exactly. The Sonnet-side passive escalation case (Haiku said "respond substantive" but Sonnet's generation emitted `[ESCALATE]` mid-flight) routes through the same fan-out for completeness.

The audit source was renamed `ella_passive_escalation_dm` → `ella_escalation_dm` since the same label now covers both paths. The dashboard's `fetchEscalationBodies` accepts both labels so historical rows continue to render. Audit rows now land one-per-recipient with `payload.path` and `payload.recipient_source` recording the origin and which CSM concern (head vs primary) the DM addresses. The `output_summary` format on escalation `agent_runs` rows shifted to `"escalated via DM; Scott=ok, Lou Perez=ok; escalation_id=<id>"` — the placeholder pattern `^escalated via DM/i` still matches so the dashboard's body-suppression path keeps working.

Test suite ran clean throughout. The execution landed in two phases: the passive-side migration + new shared module + dashboard filter shipped on commit `018bd41` (524 passing). Then the gate-(b) hard stop fired on the `prompts.py` edit — surfaced the proposed diff to Drake, wrote a partial report, pushed. Drake re-invoked `/run` to signal approval. ## Surprises and judgment calls

- **Three escalation entry points, not two.** The spec described "two paths" — reactive (`_run`) and passive Haiku-decided (`passive_dispatch._fire_escalation_dm`). Reading the code I found a third: `respond_to_passive_trigger`'s Sonnet-side escalation branch, where Haiku decided "respond substantive" but Sonnet's generation emitted `[ESCALATE]` mid-flight. It was writing an `escalations` row already but firing no DM. Under the new "all escalations DM Scott + primary CSM" rule that's a silent regression, so I wired it through `fire_escalation_dms(..., path="passive")`. Flagging because the spec doesn't call it out and a future reader might assume only two paths exist.
- **`agent_runs.status` flip from `'success'` to `'escalated'` on passive escalations.** The passive `_fire_escalation_dm` used to set `status='success'` because no escalations row was being written. With the unified write I align with the reactive path's `status='escalated'` so the dashboard's response-scope filter treats them identically. The dashboard already accepts both statuses, and the `passive_monitor` rows pass the scope filter via `haiku_decision === 'escalate'` regardless of status, so no rendering shift in practice.
- **`output_summary` format change.** Was `"escalated via DM; csm=<name>; dm_ok=<bool>"` (one CSM); now `"escalated via DM; <label1>=ok/fail, <label2>=ok/fail; escalation_id=<id>"` (N recipients + escalation id). Still starts with `escalated via DM` so the dashboard's `ESCALATION_PLACEHOLDER_PATTERN = /^escalated via DM/i` still matches and the body-suppression path keeps working. New rows render with the new shape; historical rows continue to render with the old shape (the placeholder text just isn't shown in either case — the body fetch replaces it).
- **N audit rows per escalation, not 1.** The new shape emits one `webhook_deliveries` row per recipient. The dashboard's body-fetch keys by `(slack_channel_id, triggering_message_ts)` and dedupes via overwrite — same body across recipients in a single escalation event resolves to the same key. No rendering bug, but operational queries that count audit rows now multiply by the recipient count.
- **Recipient-list dedup semantics when Scott IS the channel's primary CSM.** Collapsed to one entry tagged `source='scott'` (head-CSM concern wins in the audit trail). The alternative — tag as `primary_csm` — felt subtler and harder to reason about. Tests pin this; one-line flip if Drake wants the other shape.
- **Sonnet-side passive escalation `path` label.** I tagged it `path="passive"` even though the entry point shares code with the reactive path (Sonnet generation), because the trigger is a passive substantive (no human @-mentioned Ella). Audit queries that want to count "user @-mentioned Ella and got escalated" should filter on `path='reactive'`; "Ella passively decided to escalate at any layer" filters `path='passive'`.
- **Style examples in `ella.md` had to be rewritten.** Two of the example lines showed Ella's escalation reply containing `<@U09JYRAENPJ>`. They'd silently misalign with the new prompt rule, so I rewrote them to drop the mention. Small but worth noting because the spec didn't enumerate this.
- **Conftest patch swap, not addition.** The autouse `_block_real_slack_posts` had `agents.ella.passive_dispatch.post_message` in its monkeypatch list. After this refactor passive_dispatch no longer imports `post_message` directly, so that path is no longer a bound name — patching it would `AttributeError`. Replaced with `agents.ella.escalation_routing.post_message`. The other patches (`shared.slack_post.post_message` direct + `agents.gregory.cs_call_summary_post.post_message`) stay.
