# Ella Duplicate Webhook-Delivery Diagnostic

**Slug:** ella-duplicate-webhook-delivery-diagnostic
**Status:** in-flight
**Type:** Diagnostic (READ-ONLY investigation; produces a follow-up fix spec, not a fix)

## Context

The dedup gate shipped 2026-05-20 (`docs/specs/ella-realtime-ingest-idempotency.md`) was designed to prevent duplicate processing of the same Slack message. Production evidence shows it is not preventing the duplicate processing it was designed to prevent. The gate's PK uniqueness mechanism is functioning correctly — the dedup keys being generated by the system are non-equal across what evidence suggests is the same logical client message.

This spec investigates WHY two webhook deliveries for the same logical Slack message are landing with non-equal dedup keys. It does NOT propose a fix. The fix follows in a separate spec once the investigation has produced a confirmed root cause.

**Hard requirement on this spec:** Builder must NOT make assumptions about the cause. Builder must read the actual code, query the actual data, and report what is found. If a hypothesis emerges during the investigation, Builder must verify it with additional evidence before treating it as conclusion. Multiple competing hypotheses are acceptable as long as each is grounded in evidence; "I don't know yet, here are the next diagnostic steps" is a valid output.

## Diagnostic objectives

Builder produces evidence-based answers to the following questions. Each answer must cite specific code locations, query results, or log entries.

### Q1: What is the actual relationship between the two webhook_deliveries rows in Evidence 3?

The two rows share `payload.slack_ts` but have different webhook_ids. Possible explanations include but are not limited to:

- Two distinct Slack event types (`message`, `message_changed`, `app_mention`, `message.channels`, `message.groups`, or other) for the same underlying user message, each arriving with a different outer `event.ts`.
- Two Slack webhook subscriptions firing for the same event (`message.channels` and `message.groups` both active).
- A bug in the deterministic key construction.
- An entirely different cause not enumerated here.

Builder reports which of these (if any) is supported by the evidence and what additional evidence supports the conclusion. Builder must NOT pick the most plausible-sounding hypothesis without empirical support.

**Specific empirical step required:** Builder queries `webhook_deliveries` for these two webhook_ids and inspects the FULL `payload` jsonb (not just the `slack_ts` field). The audit payload at this layer is the parsed message record, not the raw Slack event — so the raw event shape may not be recoverable from `webhook_deliveries`. If it isn't, Builder traces upward to determine whether the raw Slack event is logged or recoverable anywhere.

**Suggested SQL:**

```sql
SELECT
  webhook_id,
  source,
  processing_status,
  received_at,
  processed_at,
  payload,
  headers
FROM webhook_deliveries
WHERE webhook_id IN (
  'slack_msg_ingest_C0AEEPVK36W_1779309969.851479',
  'slack_msg_ingest_C0AEEPVK36W_1779309983.000100'
);
```

### Q2: What outer event.ts values produced each of the two webhook_ids?

The webhook_id at line 88 of `ingestion/slack/realtime_ingest.py` is built from `slack_ts = event.get("ts")`. For the second row, the encoded ts is `1779309983.000100`. Builder reports what that outer event ts represents in Slack semantics — does the literal value `.000100` appear in real Slack event payloads, or does it suggest the outer event is something other than a top-level `message`? Resources for this investigation:

- Slack Events API documentation on event_callback shapes for `message`, `message_changed`, `app_mention`, `message.channels`, `message.groups`.
- Slack documentation on `ts` field semantics across event types.

If the Slack documentation does not provide a definitive answer, Builder reports that as the finding and suggests next steps (e.g., adding temporary instrumentation to capture raw events for the next duplicate occurrence — but Builder DOES NOT add the instrumentation in this spec).

### Q3: What does the `slack_messages` table contain for `slack_ts = '1779309969.851479'`?

`slack_messages` has a unique constraint on `(slack_channel_id, slack_ts)` (per migration 0002). If both webhook deliveries reached the `_upsert_message` call, the table will show only one row (the upsert collapses them). Builder reports:

- Whether the row exists.
- The `text` value (to confirm the message content).
- The `slack_user_id` (to confirm the client author).
- The `raw_payload` jsonb (which is the parsed event — `event_for_parser` in the realtime ingest code). This is the most accurate copy of what the parser saw.

**Suggested SQL:**

```sql
SELECT
  slack_channel_id,
  slack_ts,
  slack_thread_ts,
  slack_user_id,
  author_type,
  text,
  message_type,
  message_subtype,
  raw_payload,
  sent_at,
  ingested_at
FROM slack_messages
WHERE slack_channel_id = 'C0AEEPVK36W'
  AND slack_ts = '1779309969.851479';
```

### Q4: Are there OTHER `slack_messages` rows in the same channel with `slack_ts` near `1779309983.000100`?

The second webhook_id encodes `1779309983.000100` as its ts. If that's a real Slack-assigned ts for a different message (not the one with ts `...969.851479`), there should be a slack_messages row with that ts. If there isn't, that's a signal the outer ts isn't a real message ts.

**Suggested SQL:**

```sql
SELECT
  slack_channel_id,
  slack_ts,
  slack_user_id,
  author_type,
  LEFT(text, 100) AS text_preview,
  message_type,
  message_subtype,
  sent_at
FROM slack_messages
WHERE slack_channel_id = 'C0AEEPVK36W'
  AND slack_ts BETWEEN '1779309980' AND '1779309990'
ORDER BY slack_ts;
```

### Q5: Are there OTHER `webhook_deliveries` rows with `payload.slack_ts` shared across multiple webhook_ids, in any production channel over the trailing 7 days?

This tests whether the Trevor case is one-of or systematic. If systematic, Builder reports the scope (how many distinct messages, how many channels, what time range).

**Suggested SQL:**

```sql
SELECT
  payload->>'slack_channel_id' AS channel,
  payload->>'slack_ts' AS message_ts,
  COUNT(*) AS delivery_count,
  array_agg(webhook_id ORDER BY received_at) AS webhook_ids,
  array_agg(received_at ORDER BY received_at) AS received_times
FROM webhook_deliveries
WHERE source = 'slack_message_ingest'
  AND processing_status = 'processed'
  AND received_at > now() - interval '7 days'
GROUP BY payload->>'slack_channel_id', payload->>'slack_ts'
HAVING COUNT(*) > 1
ORDER BY delivery_count DESC, channel, message_ts
LIMIT 100;
```

### Q6: What is the empirical relationship between the outer event ts and the inner parsed slack_ts across the duplicate set?

The retry short-circuit fires on `X-Slack-Retry-Num`. If duplicate webhook deliveries are coming from Slack-initiated HTTP retries, this short-circuit should be catching them — and if it is firing, those retries wouldn't reach `ingest_message_event` at all.

This question may not be answerable from the database alone. Builder reports the access path available. If Vercel logs are not readable from Builder's environment, this question gets a "not investigated" status with a note about what would be needed to answer it.

### Q8: What is the actual structure of an `app_mention` event vs a `message` event in this codebase?

`api/slack_events.py` routes `app_mention` to a logged no-op (the comment claims a parallel `message` event handles it via the passive path). `message` events route to `_ingest_message_event`. Builder reads the Slack Events API documentation and confirms:

- Whether `app_mention` events have their own `ts` distinct from the message they're mentioning.
- Whether Slack subscriptions can produce overlapping deliveries (e.g., a `message` subscription PLUS an `app_mention` subscription, both firing for the same underlying user post).
- Whether `message.channels` and `message.groups` subscriptions can both fire for the same underlying message in a channel that is somehow visible to both.

The codebase comment in `api/slack_events.py` lines 158-167 states *"Slack fires a parallel `message` event alongside every `app_mention`"* — Builder verifies this against current Slack documentation and reports whether it's accurate and whether `app_mention` could be involved in the duplicate pattern.
