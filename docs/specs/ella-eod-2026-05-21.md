## The arc, as it happened (Builder's source material)

Builder uses this section as the factual basis for the EOD entry. Builder may read the underlying specs/reports to enrich detail but should NOT contradict this timeline — it's the authoritative account Drake confirmed in the working session.

**Starting state (2026-05-21 morning):** Production passive monitoring was paused (kill switch flipped 2026-05-19 EOD, 136 channels at `passive_monitoring_enabled=false`). Two ships had landed believed-to-resolve the 2026-05-19 misfire: the @-mention routing gate + assigned-advisor context (2026-05-20 morning) and the realtime-ingest idempotency gate (2026-05-20 afternoon). Production had been resumed late 2026-05-20 evening via `UPDATE slack_channels SET passive_monitoring_enabled = true WHERE test_mode = false`.

**The rediscovery:** Drake observed Ella double-posting in production client channels (Trevor Heck, Shivam Patel screenshots — semantically-identical ack pairs seconds apart). Initial hypothesis was that the idempotency gate from the prior afternoon wasn't working.

**The diagnostic detour:** The chat-side diagnostic worked through several wrong hypotheses before landing on the cause:
- First hypothesis: `pending_ella_responses` cron race (two cron invocations processing the same queued row). Disproven — zero rows in that table for the affected channels.
- Second hypothesis: a different response path entirely (the unanswered-flagger cron, or a reactive @-mention handler). Disproven — the unanswered flagger posts only to an internal channel; no separate reactive path was producing these.
- A query-construction error compounded the confusion: the diagnostic queried `agent_runs.trigger_metadata->>'slack_channel_id'` when the actual key is `triggering_slack_channel_id`. The wrong-key query returned zero rows, falsely suggesting the passive-monitor pipeline wasn't involved. Corrected query revealed the pipeline WAS firing — twice per logical message.
- A side-finding surfaced during the detour: Ella's own posts are classified `author_type='bot'` not `'ella'` in `slack_messages` (her user account `U0ATX2Y8GTD` isn't being recognized by the parser).

**The root cause (confirmed via formal diagnostic spec + Builder investigation):** Slack `message_changed` events (fired when a user edits a message) carry a different OUTER `event.ts` than the original `message` event for the same logical message. The idempotency gate's dedup key was built from the outer `event.ts`, so an edit produced a different dedup key, the PK collision never fired, and the full pipeline ran twice — two `agent_runs`, two acks, two escalation rows, four DMs. Production scope: 11 documented duplicate dispatches across 8 channels in ~36 hours. Zero `slack_msg_ingest_dup_*` audit rows had ever been written — the gate had never caught a true duplicate in production.

**The wrong-call acknowledgment:** the 2026-05-20 idempotency spec had explicitly anticipated this failure mode in its "What could go wrong #6" and classified it "acceptable for v1" on the assumption that Slack edits are rare in client channels. Production reality (11 dupes / 36 hours) contradicted that assumption. This was a judgment error in the prior spec, owned in the corrective work.

**The fix:** `ella-realtime-ingest-dedup-message-changed` moved the dedup-key construction from pre-parse (outer `event.ts`) to post-parse (inner `record.slack_ts`, which is stable across edits). Smoke-validated by Drake in `#ella-test-drakeonly`: posted a message, edited it, confirmed the edit produced a `slack_msg_ingest_dup_*` row with `processing_status='duplicate'` and NO second response. First time a true duplicate was caught in production-shaped testing.

**The accepted trade-off:** edits that add critical content are now suppressed (Ella sees only the pre-edit text). Accepted because the alternative (11 dupes / 36 hours) was worse for client trust, and clients can post a follow-up message to add context.

**Second resume:** after smoke validation, Drake re-ran `UPDATE slack_channels SET passive_monitoring_enabled = true WHERE test_mode = false AND passive_monitoring_enabled = false AND is_archived = false` — 136 channels back on.

**Additional ships same day:**
- `ella-unanswered-flagger-client-only-and-terse-post` — the unanswered flagger now filters to client-authored messages only (was surfacing team_member messages) and uses a terse one-line post format (was a six-line block). Drake observed it working in production (two messages came through correctly). Still pending formal gate (c) confirmation on the team_member-exclusion case.
- `ella-doc-hygiene-sweep-2026-05-21` — forward-pointer annotations added across the prior docs (so the idempotency spec/report/state-entry point forward to the corrective), plus a dead-code inventory catalogued in known-issues. Shipped.

**Operational finding (not yet logged anywhere):** the unanswered-flagger terse posts include a Slack permalink. Slack auto-unfurls the permalink into a message preview ONLY when Ella's bot is a member of the source channel. Posts from channels where the bot isn't a member render as a bare URL with no preview. This is Slack-side unfurl behavior, not a code bug — but it's an operational note: to get previews, ensure Ella's bot is added to client channels.

**Architectural conversation (tabled):** Drake raised that Ella may need a rehaul — passive monitoring adds too much complexity, she lacks the content coverage to help deeply (KB is call transcripts/summaries/reviews only, no curriculum/GHL/offer docs), and her 15-message context window limits depth. Four product-shape options were discussed and tabled for a future conversation (likely with Nabeel): (1) CSM copilot — Ella becomes a team-facing tool, never posts in client channels; (2) pure escalation router + digest — reads everything, never responds in-channel, only routes; (3) curriculum FAQ bot scoped tightly to high-similarity KB hits; (4) status quo with continued patching. No decision made; flagged so it resurfaces.
