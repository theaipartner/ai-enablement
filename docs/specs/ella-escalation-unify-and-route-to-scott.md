# Unify Ella escalation paths (reactive + passive) and route to Scott + primary CSM

**Slug:** ella-escalation-unify-and-route-to-scott
**Status:** in-flight

## Context

Today Ella has two escalation paths that behave differently:

- **Reactive (`agents/ella/agent.py`)** — when Sonnet emits `[ESCALATE]`, the agent posts a client-facing ack in channel that includes an `<@advisor_slack_user_id>` mention of the channel's primary CSM. Persists an `escalations` row via `agents/ella/escalation.py:escalate`.
- **Passive (`agents/ella/passive_dispatch.py:_fire_escalation_dm`)** — when Haiku returns `"escalate"`, the agent posts nothing in channel and fires a backend DM to the channel's primary CSM with a Slack permalink + truncated Haiku reasoning. Audited via `webhook_deliveries.source='ella_passive_escalation_dm'`. **Does NOT write to the `escalations` table.**

Drake (head of AI Innovation) and Scott (head CSM) want every escalation routed to Scott as a backend DM in addition to the assigned primary CSM, with the two paths behaving identically below the surface:

1. **Reactive path keeps its in-channel client-facing ack** — but with the advisor `<@...>` mention dropped. The client sees acknowledgment that someone is handling it; no advisor gets pinged in the channel. (Passive path is unchanged — fully silent in channel.)
2. **Both paths fire backend DMs to TWO recipients: Scott + the channel's primary CSM.** Two `post_message` calls per escalation.
3. **Both paths write an `escalations` row.** Passive currently doesn't; this closes the persistence gap so `/ella/runs` + future ops reporting see both shapes identically.

Scott is identified via a new env var **`ESCALATION_RECIPIENT_SLACK_USER_ID`**. Reasoning logged in § Decisions below. Safer fallback: if the env var is unset, only the primary CSM is DMed — escalations still work, Scott is just silently not CC'd. No exceptions raised on missing env var.

## Decisions baked in (do NOT re-litigate)

- **Scott identification: env var `ESCALATION_RECIPIENT_SLACK_USER_ID`.** Not a `team_members` flag, not a config table. Reasoning: matches the existing pattern for globally-configured Slack identities (`SLACK_CS_ACCOUNTABILITY_CHANNEL_ID`, `SLACK_CS_CALL_SUMMARIES_CHANNEL_ID`), zero migration, zero schema change, and if we ever want rotation or multiple recipients the migration to a flag/table is small and contained.
- **Safer fallback on missing env var: primary CSM only.** If `ESCALATION_RECIPIENT_SLACK_USER_ID` is unset, escalations still fire — just only to the primary CSM. Log a warning. Do NOT raise. The current escalation behavior is the floor.
- **Two DMs per escalation, not one.** Scott + primary CSM. If they happen to be the same person (Scott is some client's primary CSM), de-duplicate so only one DM lands. Use a `dict.fromkeys([scott_id, primary_csm_id])` style dedup or equivalent.
- **Reactive path keeps the client-facing ack, drops the advisor mention.** The client still sees Ella acknowledge the handoff. This requires a prompt edit in `agents/ella/prompts.py` so Sonnet no longer writes `<@U...>` in escalation responses.
- **Both paths write `escalations` rows.** The reactive path already does this via `escalate()`. The passive path needs to start calling it. The `context` dict on the passive `escalations` row should mirror the reactive one as closely as possible: `query_text` = triggering message text, `ella_response` = empty string (passive never posts), `handoff_reasoning` = Haiku reasoning, `client_id`, and any passive-specific metadata (Haiku decision, KB chunks summary) the reviewer might find useful.
- **DM body shape:** Same body, same shape, for both Scott and the primary CSM. Same body shape, same shape, for both reactive and passive — they should be visually indistinguishable when read in Slack. Use the existing passive DM body shape as the template:
  - `":eyes: Worth a look — <slack-permalink-to-triggering-message>\n_Ella escalated this. Reasoning: <reasoning, truncated to 200 chars>_"`
  - For the reactive path, the "reasoning" is the `handoff_reasoning` Sonnet wrote after `[ESCALATE]`. For the passive path, it's the Haiku reasoning. Same field name, different source.

## Implementation plan (specs the work, doesn't write the code)

### 1. New shared helper: `agents/ella/escalation_routing.py` (new file)

Houses the two new shared concerns: recipient resolution and DM fan-out. Pulling this out of both `agent.py` and `passive_dispatch.py` is the only way the two paths can stay truly identical going forward.

Public surface:

```python
def resolve_escalation_recipients(primary_csm: dict | None) -> list[dict]:
    """Returns a deduplicated list of recipients to DM.
    Each recipient is {"slack_user_id": str, "label": str, "source": "scott" | "primary_csm"}.
    """

def fire_escalation_dms(
    *,
    recipients: list[dict],
    slack_channel_id: str,
    triggering_message_ts: str,
    reasoning: str,
    path: str,  # "reactive" | "passive"
) -> list[dict]:
    """Fires one DM per recipient. Returns a list of per-recipient results.
    Audited under webhook_deliveries.source='ella_escalation_dm' (NOTE: renamed from
    'ella_passive_escalation_dm' to cover both paths — see Migration section below).
    """
```

`resolve_escalation_recipients` reads the env var, looks up `team_members.full_name` for the Scott label (best-effort — falls back to "Scott" string if lookup fails or is slow), and returns the merged + dedup'd list. Order: Scott first, primary CSM second.

`fire_escalation_dms` builds the body once (it's the same for all recipients) and fires N `post_message` calls. Per-recipient audit rows so a missing-bot-from-DM-channel kind of failure on one recipient doesn't hide which one failed.

### 2. Audit-source rename: `'ella_passive_escalation_dm'` → `'ella_escalation_dm'`

The single audit source label now covers both paths. Forward-only — historical rows under the old label stay queryable; the dashboard's `lib/db/ella-runs.ts:fetchEscalationBodies` already filters by source, so update that filter to accept both labels during the transition.

Add a comment in `passive_dispatch.py` (or wherever the constant moves) noting the historical name. **Don't** retroactively update old rows — they're audit history.

### 3. Reactive path changes (`agents/ella/agent.py`)

- After `escalate()` writes the `escalations` row, call `fire_escalation_dms()` with `path="reactive"`. `reasoning` is the `handoff_context` Sonnet emitted after `[ESCALATE]`.
- `client_text` (what the client sees in channel) is still returned to the Slack handler — but it should no longer contain an `<@U...>` advisor mention. The prompt change (step 5 below) is what causes this.
- The `EllaResponse.escalation_id` field already exists; keep it. The Slack handler still posts `client_text` to the channel.

### 4. Passive path changes (`agents/ella/passive_dispatch.py`)

## What could go wrong

Think this through yourself before starting. Some seeds:

- **Prompt edit underspecifies the new behavior.** Sonnet starts dropping the ack entirely, or writes something cold like "Escalated.". The client experience degrades to silence-with-extra-steps. Mitigation: write the prompt instruction with at least one example of an acceptable ack, and verify with a manual test.
- **Scott has DMs disabled or Ella isn't authorized to DM him.** The first DM in production fails with `cannot_dm_bot` or similar. Mitigation: the `post_message` helper already returns `ok=False` rather than raising; the audit row captures the slack_error; the second DM (to primary CSM) still fires.
- **The dedup logic strips one recipient when Scott IS a primary CSM somewhere.** That's correct behavior — Scott shouldn't get two DMs about the same escalation just because he happens to be the assigned CSM too. Tests should cover this case explicitly.
- **The audit-source rename breaks the `/ella/runs` escalation body rendering for historical rows.** Mitigation: the filter accepts both old and new labels; no historical row data changes.
- **Sonnet's `[ESCALATE]` token detection regresses** because the prompt change inadvertently weakens Sonnet's escalation-marker emission discipline. Mitigation: the prompt change should ONLY touch the format of the client-facing ack, NOT the escalation criteria or marker instructions. Diff with care.
- **Env var name collision or typo.** `ESCALATION_RECIPIENT_SLACK_USER_ID` is the canonical name; don't rename it mid-spec.
- **The new shared module creates a circular import.** `agents/ella/escalation.py` already imports from `shared.hitl`; the new `agents/ella/escalation_routing.py` imports from `shared.slack_post` and possibly from `agents/ella/escalation` if you decide to merge them. Be careful; this is a foot-gun in the Ella module tree historically.
